# PM One

## Design Context

### Users
PM One is an internal staff dashboard used by event organizers (Panorama Media + partner agencies) to manage Indonesian B2B exhibitions: brands/exhibitors, orders, rundown, promotion posts, hotels & reservations. Primary users are staff (master/admin roles) doing operational work during business hours from desktop. Secondary users: project members + exhibitor PICs.

### Brand Personality
**Focused. Direct. Quietly confident.** Tools-first product — staff are here to do work, not be entertained. The interface earns trust by being predictable, readable, and not wasting attention. Closer to Linear or Vercel dashboard than Notion or Stripe marketing.

### Aesthetic Direction
**Light mode default, dark mode supported.** Modern admin SPA (Nuxt 4 + shadcn-vue + Tailwind v4). Restrained surfaces, generous information density when justified, tight typography. No glassmorphism, no rainbow gradients, no decorative sparklines. Each section/card is a clean rectangle with a thin border.

### Design Principles (binding)

These come from accumulated feedback in `~/.claude/projects/-Users-nextifier-Herd-pmone/memory/MEMORY.md` and are non-negotiable:

1. **Typography**
   - ALWAYS `tracking-tight` on body. Use `tracking-tighter` for `text-xl` and above.
   - Max font weight is `font-semibold`. **Never** `font-bold` or `font-extrabold`. Use `font-medium` for buttons, links, small section titles.
   - Avoid `text-xs` (and smaller) standalone on larger screens. Use `text-xs sm:text-sm` for fine print.
   - Page titles: use the project's `page-title` class. Section descriptions: `page-description`.

2. **Color**
   - Use CSS variable tokens (`bg-success`, `bg-warning`, `bg-destructive`, `bg-info`, `bg-muted`, `bg-primary`, `text-muted-foreground`, etc.) — **never** raw Tailwind palette (`bg-green-*`, `bg-yellow-*`, `bg-red-*`, `bg-blue-*`).
   - Status tones: success / warning / destructive / muted.

3. **Spacing & Layout**
   - Use shared frame primitives (`frame`, `frame-header`, `frame-title`, `frame-panel`) for editable forms.
   - Use shared TabNav, TableData, DialogResponsive primitives — don't roll your own.
   - Section spacing typically `space-y-6` on page, `space-y-4` inside frame, `space-y-2` inside form group.

4. **Interaction**
   - **NEVER** `confirm()` / `alert()` / native `prompt()`. Always `DialogResponsive` for confirmations.
   - Optimistic UI for: toggles, quick add, delete, inline title edit.
   - Toasts: `toast.success/error` from `vue-sonner`. Keep description short, factual.
   - DialogResponsive auto-renders as Dialog on desktop, BottomSheet on mobile — never wrap content in `<form>` that submits the parent page.

5. **Writing**
   - **Never** em-dash (—). Use `-` or comma.
   - Required field indicators are auto-rendered via global CSS rule (`#layout-app label:has(+ input:required)::after`). **Never** add manual `<span class="text-destructive">*</span>` markers.
   - Section / page titles in Title Case. Helper text in sentence case.
   - i18n: literal English strings in admin (Staff Dashboard). i18n only in pmone-events, Dashboard Exhibitor, and `components/ui` (synced with pmone-events).

6. **Forms**
   - Auto-slug from name fields. Use `Select` for city/country (from `frontend/app/data/`).
   - Tags via `spatie/laravel-tags` (TagsInput primitive).
   - Boolean toggles: shadcn-vue `Switch` with `v-model` (reka-ui v2 — uses `modelValue`, not `:checked`).
   - Settings persisted to `model.settings` / `model.more_details` JSON. Don't add new columns unless truly needed.
   - Hide fields that aren't important — minimize information overload.

7. **Anti-patterns to avoid**
   - No hotel/bed icons as placeholder fallbacks — use neutral gradient (`from-muted to-muted/40 bg-gradient-to-br`).
   - No `border-left: 3px solid` accent stripes.
   - No gradient text.
   - No nested cards.
   - No `font-bold`.

### Tech & Code Conventions
- Frontend: Nuxt 4 + Vue 3 SFC + Tailwind CSS v4 + shadcn-vue + reka-ui v2 + @vueuse/core. Component imports usually auto-imported by Nuxt.
- Backend: Laravel 12, response cached for public endpoints, ResponseCache tags clearable per resource.
- Build is heavy — **never** run `nuxi build` / `nuxi typecheck` / `npm run build` from the assistant terminal. Verify in browser via `localhost:3000`.
- API auth via Sanctum cookie. Use `useSanctumClient()` for client-side fetches inside child routes (avoid hydration mismatch with `useLazySanctumFetch` in nested children).
- Local URLs: `localhost:3000` (frontend), `localhost:8000` (backend). Never use `pmone.test`.
- Git: assistant must **never** `git add`, `git commit`, `git push`. User runs all git.
